### Gap 1: NVIDIA OpenShell vs AWS AgentCore Cedar — Technical Comparison

**OpenShell (NVIDIA):**
- Policy format: **Declarative YAML** (not Cedar/NL)
- Runtime stack: K3s Kubernetes cluster inside a single Docker container
- Four policy domains (defense in depth):
  1. **Filesystem** — prevents reads/writes outside allowed paths (locked at sandbox creation)
  2. **Network** — blocks unauthorized outbound connections (hot-reloadable at runtime)
  3. **Process/syscall** — blocks privilege escalation and dangerous syscalls (locked at sandbox creation)
  4. **Inference routing** — reroutes model API calls to controlled backends (hot-reloadable)
- Enforcement level: **OS/infrastructure layer** — kernel-level enforcement via syscall blocking
- SDK: `uv tool install -U openshell` (CLI) + `pip install nvidia-nat` (NeMo Agent Toolkit)
- Google integration: Google is listed as a *security partner* (not a cloud infra integration) — OpenShell is compatible with Google Security tools via their security collaboration
- NeMo Agent Toolkit supports **Google ADK** natively (plus LangChain, LlamaIndex, CrewAI, Semantic Kernel)

**AWS AgentCore Policy (Cedar):**
- Policy format: Cedar (attribute-based) + natural language intent
- Enforcement level: **Semantic/application layer** — which agent can call which tool, with what permissions
- Sits outside agent code; governs tool authorization at the API/service boundary

**Key insight — they are different layers, not direct substitutes:**
- AgentCore = semantic permission layer (agent→tool authorization, like IAM for agents)
- OpenShell = infrastructure enforcement layer (what the agent process can do at OS level, like seccomp + network policy + filesystem jailing)
- A hardened stack needs both. GCP has neither equivalent today.
- Vertex AI Agent Identity + Cloud IAM covers part of the AgentCore layer but has no OpenShell-equivalent runtime sandbox.

### Gap 3: Multi-Agent Teams Underperform Single Expert by 37.6% — When and Why

**Paper:** "Multi-Agent Teams Hold Experts Back" (arXiv:2602.01011)

**The 37.6% figure:** Maximum observed performance loss, not average. Tested across human-inspired and frontier ML benchmarks.

**Root cause — "integrative compromise":**
- Teams do NOT simply fail to identify the expert — they fail even when **explicitly told who the expert is**
- Teams engage in weighted averaging of all opinions instead of deferring to the identified expert
- Result: expert knowledge is diluted by non-expert views
- Counterintuitive: this same consensus behavior **improves robustness against adversarial agents** — so it's a fundamental tradeoff, not a pure bug

**Conditions that make it worse:**
- Larger team sizes → worse expertise utilization (more averaging = more dilution)
- Self-organizing teams (no explicit routing architecture)
- Democratic/voting-style consensus protocols

**Vertex AI relevance — architecture implication:**
- Self-organizing multi-agent topologies (fan-out, vote, synthesize) will likely underperform on expert tasks
- Better approach: explicit supervisor routing with hard specialization (agent receives only the sub-tasks it's designed for) rather than team deliberation
- Vertex AI Agent Engine's ADK supports explicit subagent delegation — use that pattern, not collaborative consensus
- AdaptOrch (arXiv:2602.16873, also in ledger) addresses dynamic topology selection — may partially solve this by selecting the right topology per task type
