## Deep Dive: Tobira.ai — Agent-to-Agent Business Discovery & Negotiation

**Researched:** March 28, 2026

### What Tobira Actually Is
Tobira.ai is a free, open agent-to-agent networking platform. Core concept: every AI agent gets a public @handle address in a shared network. Agents discover other agents, hold autonomous conversations about goals/fit/working style, and ping their human owners only when both sides confirm mutual interest. Launched Product Hunt March 23, 2026 — hit #1 Product of the Day with 682 upvotes, 153 comments.

Tagline: "Give your AI agent an @address in the AI agent network."
Product Hunt: https://www.producthunt.com/products/tobira-ai
Website: https://tobira.ai/

### Architecture: How Agents Discover & Connect

1. **Identity Layer**: Each agent receives a public @handle. Anti-cybersquatting enforced — agents must have "real context" to claim a handle (not just name-squatting).

2. **Discovery Mechanism**: Not technically documented publicly. The platform uses match-based pairing (like a dating-app model for agents) — agents are matched based on goals, context, and declared capabilities. No public API spec for how matching works.

3. **Communication Layer**: Built on MCP (Model Context Protocol) servers. Agents communicate via "API and MCP for other agents." Compatible explicitly with:
   - OpenClaw (Claude Code extension)
   - Claude Cowork
   - Open API + MCP for any other agent framework

4. **Privacy Controls**: Users set anonymous vs. public disclosure. Contact details hidden until *both* parties grant mutual approval (double opt-in model). Prevents spam and cold outreach.

5. **Conversation Flow**: Agent-to-agent conversations are real dialogues — "about goals, fit, and working style." Not just capability queries. Humans only get notified when both agents confirm the match is real.

6. **Infrastructure**: Built with Claude Code, deployed on Railway.

## A2UI Deep Dive (Researcher Request — follow-up)

**Researched:** March 28, 2026

### What A2UI Is

A2UI (Agent-to-User Interface) is a **declarative JSON format** for agents to describe and stream rich UI components to a client renderer. Created by **Google**, announced December 15, 2025. Open-source under **Apache 2.0**.

- **Spec site:** https://a2ui.org/
- **GitHub:** https://github.com/google/A2UI (13.7k stars, 1k forks, 51 contributors, 519 commits)
- **Blog announcement:** https://developers.googleblog.com/introducing-a2ui-an-open-project-for-agent-driven-interfaces/
- **Current stable spec:** v0.8 (Public Preview — December 2025)
- **Draft spec:** v0.9 (in progress)

### Core Design Philosophy

Three problems A2UI solves:
1. **Contextual UI generation**: agents generate interfaces appropriate to context, not just text
2. **Safe cross-trust-boundary rendering**: multi-agent systems where a sub-agent can't directly manipulate UI
3. **Framework-agnostic portability**: same agent output renders natively on Web, Flutter, Angular, SwiftUI

Key security model: **declarative data, not code**. Client maintains a "catalog" of pre-approved components (Button, Card, TextField, etc.). Agent can only *reference* catalog items — it cannot inject arbitrary code. Zero XSS/injection attack surface.

LLM-friendly architecture: flat component lists with ID references (adjacency list model), designed for incremental generation and streaming partial updates.

### Format

**JSON / JSONL** — specifically JSONL for streaming over SSE. The agent outputs a sequence of JSON envelope messages, each on a separate line. No protobuf. No binary format.

Schemas split in v0.9 into:
- `common_types.json`
- `basic_catalog.json`
- `server_to_client.json`
- `client_to_server.json`

### Transport

**Transport-agnostic by design**. The protocol only defines the JSON message schema and semantic contract. Supported transports:

| Transport | Use Case |
|-----------|----------|
| **SSE (Server-Sent Events) + JSONL** | Primary, web streaming |
| **WebSocket** | Bidirectional real-time sessions |
| **A2A (Agent2Agent)** | Sub-agent to orchestrator to client pipeline |
| **AG-UI** | Agent-to-user interaction layer |
| **MCP** | Delivered as tool outputs or resource subscriptions |
| **REST** | Simple use cases without streaming |

### Message Types — v0.8

**Server-to-Client (4 envelope types):**

1. **`surfaceUpdate`** — delivers component definitions to a named surface
   - Fields: `surfaceId`, `components[]` (each with `id` + `component` map)

2. **`dataModelUpdate`** — manages dynamic state separate from UI structure
   - Fields: `surfaceId`, `path` (optional JSON path), `contents[]` (key/value entries with typed values: `valueString`, `valueNumber`, `valueBoolean`, `valueMap`)

3. **`beginRendering`** — signals client to start rendering, specifies root component
   - Fields: `surfaceId`, `root` (component ID), `catalogId` (optional)

4. **`deleteSurface`** — removes a surface and all its components
   - Fields: `surfaceId`

**Client-to-Server (2 types, sent via A2A back-channel):**

1. **`userAction`** — user interaction events (button click, form submit)
   - Fields: `name`, `surfaceId`, `sourceComponentId`, `timestamp` (ISO 8601), `context` (resolved data bindings)

2. **`error`** — client-side error reporting (flexible structure)

### Message Types — v0.9 Changes (Draft)

v0.9 renames the envelopes (no new types, just renamed):
- `createSurface` (was implicit in beginRendering)
- `updateComponents` (was surfaceUpdate)
- `updateDataModel` (was dataModelUpdate)
- `deleteSurface` (unchanged)

v0.9 also adds:
- `sendDataModel` boolean flag on `createSurface` — enables client to transmit full data model state alongside user actions
- `a2uiClientCapabilities` and `a2uiClientDataModel` now passed via transport metadata (not as protocol messages)
- Philosophical shift: "prompt-embedded schema" — LLM generates JSON matching examples rather than strictly constrained structured output. Requires post-generation validation.

v0.9 is draft status — v0.8 is recommended for production.

### Data Binding System

Components reference application state via JSON path notation:
- `"value": { "path": "/booking/date" }` — binds component to data model path
- Literal values: `literalString`, `literalNumber`, `literalBoolean`, `literalArray`
- Combining path + literal initializes state AND binds it simultaneously

### Container/Children System

Two ways to define parent-child relationships:
- **`explicitList`** — array of component IDs for static children
- **`template`** — dynamic list with `dataBinding` + `componentId` for data-driven repetition

### Catalog Negotiation

Server declares: `supportedCatalogIds`, `acceptsInlineCatalogs`
Client declares: `a2uiClientCapabilities` object sent via transport metadata (v0.9)

This enables multi-agent scenarios where orchestrators negotiate what UI capabilities a renderer supports before delegating to a sub-agent.

### Launch Partner Ecosystem

- **CopilotKit / AG-UI**: day-zero compatibility, co-launch partner
- **Opal** (Google's AI mini-app platform): uses A2UI for dynamic interfaces
- **Gemini Enterprise**: custom UIs for enterprise agents
- **Flutter GenUI SDK**: A2UI as the declaration format for generative mobile UIs
- Multiple internal Google teams standardizing on A2UI

### A2UI vs AG-UI — Technical Comparison

| Dimension | A2UI (Google) | AG-UI (CopilotKit) |
|-----------|---------------|---------------------|
| **Role** | UI declaration format (what to render) | Transport + connection layer (how to send it) |
| **Layer** | UI spec / payload format | Protocol / runtime connection |
| **Format** | JSONL (declarative component tree) | Event stream (bidirectional, SSE-based) |
| **Direction** | Agent → Client (primarily) | Bidirectional (agent ↔ UI) |
| **Creator** | Google (Apache 2.0) | CopilotKit (MIT) |
| **Spec status** | v0.8 stable, v0.9 draft | Independent spec, growing adoption |
| **Governance** | Google-led (open source) | CopilotKit-led (community, not in AAIF) |
| **Security model** | Pre-approved catalog, no code exec | Event-based, no declared security model |
| **Relationship** | AG-UI *transports* A2UI payloads | AG-UI is A2UI-compatible transport |
| **AWS support** | Not yet (AWS adopted AG-UI, not A2UI) | AWS Bedrock AgentCore (GA March 2026) |

**Key architectural insight:** A2UI and AG-UI are **complementary, not competing**. A2UI defines *what* the UI looks like (the payload). AG-UI defines *how* it travels and *how* user events return to the agent (the protocol channel). CopilotKit positioned this clearly: "Agents generate UI using A2UI, AG-UI transports that specification from agent to application."

### GCP Implication

A2UI is a Google-originated spec but lives in open-source (github.com/google/A2UI). This is GCP's play in the UI protocol layer — they own the spec where AWS owns the runtime (AG-UI in AgentCore). GCP's advantage: A2UI integration with Gemini Enterprise and Flutter GenUI SDK creates a closed loop for Google-ecosystem agents. Gap: AWS has adopted AG-UI but not A2UI — if A2UI doesn't get adopted outside Google, the protocol war could split the ecosystem.
