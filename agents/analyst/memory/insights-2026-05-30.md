5. MCP drop-in: ARI wraps MCP tool calls transparently (matches AgentTrust's MCP integration approach)

Infrastructure: Agent Engine intercept hooks already exist (Trace Viewer proves events fire); net new is policy evaluator + decision engine (~2-3 sprints). WIF + Cloud Logging already in stack.

**Impact:** Directly converts the 92% "concerned" enterprises into GEAP customers — this is the missing governance primitive for regulated workloads. 💡 **Memory Bank GEM Compliance — Current Architecture Provably Incorrect for Multi-Agent**

**Evidence:**
1. [Is Agent Memory a Database? (arXiv:2605.26252)](https://arxiv.org/abs/2605.26252) by Orogat & Mansour — Introduces GEM (General Event Memory) framework with 4 state-level memory operators. **Formal proof: no RDBMS or vector store satisfies GEM correctness conditions for agent memory workflows.** GEM requires state-event integrity that relational and vector architectures structurally cannot provide.
2. [Governed Collaborative Memory (arXiv:2605.04264)](https://arxiv.org/abs/2605.04264) — 4 governance regimes for multi-agent shared memory: Private, Shared-Read-Only, Shared-Collaborative, Public. Current Memory Bank has no governance layer — all agents sharing a namespace see the same data with no policy differentiation.
3. [Memory Bank Survey (2605.06716)](https://arxiv.org/abs/2605.06716) — Memory governance convergence: all major 2026 agent deployments moving toward governed memory with read/write access control per agent.
4. Hindsight (vectorize-io) — 91.4% LongMemEval vs 60.2% GPT-4o baseline using TEMPR retrieval (4 parallel strategies + cross-encoder rerank). Memory Bank uses semantic search only.

**GCP Today:** Vertex AI Memory Bank is a managed vector store with semantic retrieval and TTL-based expiration. It does NOT have: GEM state operators, governance regimes, TEMPR retrieval, confidence scoring, belief evolution, or multi-agent access control. The paper explicitly proves that this architecture is insufficient for the correctness requirements of multi-agent agent memory workflows.

**Gap — fundamental:** Memory Bank v1 is a vector store with semantic search. GEM proves this is architecturally insufficient — not "less featured" but structurally incapable of providing correct memory behavior in multi-agent workflows where write ordering, read consistency, and agent-scoped access matter. AWS Memory (metadata + semantic) is ahead on filtering; neither AWS nor Azure has GEM-compliant memory. **First-mover opportunity for GCP to define the standard.**

**Recommendation — Memory Bank v2 (GEM-Compliant):**
```python
# Memory Bank v2 API
memory = MemoryBank(
    mode="governed_collaborative",  # or private / shared_read_only / public
    agents=[agent_a_id, agent_b_id, agent_c_id],
    write_policy=MemoryWritePolicy(
        gate="write_time_salience",  # per arXiv:2603.15994 — write-time filtering outperforms read-time
        min_salience=0.7,
        dedup_threshold=0.85,
    ),
    retrieval=MemoryRetrieval(
        backend="TEMPR",   # 4-strategy parallel + cross-encoder rerank; fallback to semantic
        confidence_threshold=0.6,
    ),
    governance=MemoryGovernance(
        regime="shared_collaborative",  # agents can read/write each other's memories within scope
        agent_scope=AgentScope.TASK,    # memories expire when task graph completes
        audit=True,
    )
)
```
Architecture:
1. GEM state operators as first-class API: `memory.observe()`, `memory.recall()`, `memory.forget()`, `memory.validate()` — replacing raw upsert/query
2. Governance layer: per-agent-identity read/write permissions on memory namespace; scoped by task or session
3. TEMPR retrieval: parallel semantic + BM25 + entity graph + temporal → cross-encoder reranked; semantic remains default; TEMPR opt-in for complex workflows
4. Write-time salience gate: classify importance at write time (not at read time — proven superior by arXiv:2603.15994)
5. Migration path: Memory Bank v1 compatible; v2 activated via `mode` parameter

Infrastructure: Memory Bank is a separate microservice; GEM operators = API wrapper layer; governance = WIF integration + per-call policy check; TEMPR = additional retrieval backends (~1 quarter to build properly).

**Impact:** First GEM-compliant cloud agent memory. Positions GEAP ahead of AWS (metadata-only filtering) and Azure (no multi-agent shared memory governance). Direct answer to the paper that just proved existing platforms are architecturally wrong. For enterprise multi-agent workflows (customer support teams, coding agent teams, research teams), correct memory behavior is mission-critical — incorrect memory causes agent hallucinations that compound across the team. This is a durable product differentiation.

**Priority:** HIGH — paper published May 25 (5 days ago); no cloud platform has responded; GCP named specifically as a gap; 2-3 month window before competitors respond

💡 **GEAP Process Audit Layer — The 9.3% Invisible Quality Gap**

**Evidence:**
1. [OpenClawBench Outcome-Process Gap (arXiv:2605.29253)](https://arxiv.org/abs/2605.29253) — **9.3% of trajectories that pass outcome evaluation are process-anomalous** (correct output reached via incorrect reasoning, skipped steps, or constraint violations). Standard eval frameworks (pass/fail) miss all of these. GEAP evaluation layer does outcome eval only.
2. [Taxonomy and Consistency of Safety Benchmarks (arXiv:2605.16282)](https://arxiv.org/abs/2605.16282) by Li, Fung et al. — 40 benchmarks analyzed, Kendall W=0.10 (essentially random concordance). **Safety claims backed by a single benchmark are methodologically unsound** — the benchmarks themselves contradict each other at near-random rates.
3. [Reframing Agent Security as Agent-Human Interaction (arXiv:2605.24309)](https://arxiv.org/abs/2605.24309) — Industry-academia inversion: what academia studies (intent anchoring, trust labeling) differs from what production deploys (policy spec, runtime approval). Production agents in regulated industries need process guarantees, not just output guarantees.
4. AWS AgentCore Evaluations — 13 built-in quality evaluators across correctness, helpfulness, tool selection accuracy, etc. GEAP has outcome eval; AWS has 13-dimensional scoring.

**GCP Today:** Agent Engine has an Evaluation Layer (simulate user interactions, score responses). Trace Viewer shows execution paths. But there is NO process-anomaly detection — no way to flag trajectories that pass output eval but used wrong tools, violated constraints, or reasoned incorrectly to reach a correct answer. Safety benchmark claims (single benchmark) are methodologically unsound per the paper.

**Gap:** 9.3% invisible failure rate is a regulatory nightmare for healthcare, finance, and legal agents. A medical diagnosis agent that reaches the right answer via wrong clinical reasoning is dangerous even if the output is correct. GEAP audit-level customers (HIPAA, SOX, EU AI Act) need process guarantees. Currently, GEAP evaluation is output-only + trust-the-benchmark — both insufficient.

**Recommendation — GEAP Agent Process Auditor:**
```python
# Agent Engine evaluation config
eval_config = AgentEvalConfig(
    outcome_eval=True,    # existing
    process_audit=ProcessAudit(
        enabled=True,
        checks=[
            ProcessCheck.TOOL_USE_COMPLIANCE,       # only approved tools used in approved order
            ProcessCheck.CONSTRAINT_ADHERENCE,      # no policy violations in intermediate steps
            ProcessCheck.REASONING_GOAL_ALIGNMENT,  # intermediate steps align with declared goal
            ProcessCheck.ANOMALY_DETECTION,         # flag trajectories with outcome-process divergence
        ],
        anomaly_threshold=0.08,  # alert if >8% of sessions are process-anomalous
        benchmark_suite=["geap-safety-v1", "geap-reasoning-v1"],  # multi-benchmark to avoid Kendall W fragility
    ),
    audit_export=CloudLogging(compliance_format="json-ld"),  # EU AI Act Article 12 traceability
)
```
Architecture:
1. Process scorer: runs on execution traces already captured by Trace Viewer; computes 4-dimension process score (tool compliance, constraint adherence, goal alignment, anomaly flag)
2. Multi-benchmark safety scoring: GEAP-native benchmark suite covering 3+ dimensions (avoids single-benchmark Kendall W=0.10 fragility); Cloud Monitoring dashboard shows benchmark concordance across runs
3. Anomaly alert: when outcome passes but process score is below threshold → alert Cloud Monitoring + pause agent for human review
4. Compliance export: structured JSON-LD audit records to Cloud Logging (EU AI Act Article 12, HIPAA audit trail requirements)
5. GEAP-bench: publish GEAP's own multi-benchmark suite as open eval standard (positions GCP as safety evaluation standard-setter, not just platform)

Infrastructure: Trace Viewer already captures the trajectory data; process scoring = lightweight classifier on trace structure (~1 sprint); multi-benchmark suite = eval engineering work (~2 quarters for comprehensive coverage).

**Impact:** Closes the 9.3% invisible quality gap that regulators will eventually require. First cloud platform with process-quality audit (vs output-only). Positions GEAP as the compliance-ready agent platform — direct sales argument for healthcare/finance/legal enterprises where "correct output via wrong reasoning" is a liability, not just a quality concern. Publishing GEAP-bench as open standard captures mindshare in the fragmented evaluation space.

**Priority:** HIGH — regulators (EU AI Act, HIPAA, SOX) are beginning to require process-level audit; 9.3% gap is a measurable liability; GEAP currently has no answer to auditor scrutiny of agent reasoning
