## Message 1 of 5

💡 **GEAP Managed Agent Harness — Zero-Orchestration-Code Agent Loop**

**Evidence:** AWS AgentCore Managed Harness launched in preview today ([AWS announcement](https://aws.amazon.com/about-aws/whats-new/2026/04/agentcore-new-features-to-build-agents-faster/)). Specify model + system prompt + tools → AgentCore runs the full reasoning/tool/response loop. 3 API calls. No orchestration code. MicroVM per session, model-agnostic, filesystem persistence. This is "Heroku for agents." Enterprise developers choosing between "write an ADK agent in Python" and "call 3 APIs" will choose the latter for 80% of use cases.

**GCP Today:** ADK v2 (code-first, Python/Java/Go/TS). Workspace Studio (visual, no-code). Managed Agents API (hosting). Nothing wires model+prompt+tools into a ready-to-run agent loop via REST API without writing any orchestration code.

**Gap:** No `AgentHarness` API surface in GEAP. Enterprise teams who want to "try an agent" in a day face ADK setup friction. AWS is building the top-of-funnel for agent adoption with no-code harness. GCP ADK is the right tool for power users; GEAP needs a simpler on-ramp.

**Recommendation:** Create `POST /v1/agents` GEAP REST API:
```json
{
  "model": "gemini-3-5-flash",
  "system_prompt": "You are a customer support agent...",
  "tools": ["search", "calendar", "email"],
  "memory_enabled": true,
  "session_timeout_hours": 4
}
```
Returns `{agent_id}`. Then `POST /v1/agents/{id}/runs` with `{user_message}` streams the response. Backed by existing ADK runtime internally. Expose "Quick Agent" wizard in GEAP console (visual → generates harness config). No infrastructure code required.

**Impact:** Closes the #1 UX gap vs AWS today. Expands GEAP's addressable developer base from "ADK developers" to "any developer with a use case." Enterprise trial → production conversion accelerates.

**Priority:** HIGH — AWS shipped this; GCP should ship parity within one quarter

## Message 2 of 5

💡 **Agent Runtime OS Security Rings — gVisor Ring-Based Privilege Model**

**Evidence:** Two converging signals today:
1. [arXiv:2605.14932](https://arxiv.org/abs/2605.14932) (Pirch et al.) — formal mapping of OS privilege rings to agent runtimes. Finding: **Agent Engine containers = ring-0 (no privilege separation)**. AWS Firecracker microVMs model ring-based isolation natively. A compromised agent in ring-0 can reach any resource the container's service account accesses.
2. Google Threat Intel confirmed [Claude Code used in extortion campaign against 17 orgs](https://www.euronews.com/next/2026/05/27/hackers-are-using-ai-to-find-security-flaws-no-scanner-can-catch-google-warns). Real agents, real exploitation, happening now.

**GCP Today:** Agent Engine runs in Cloud Run containers with gVisor. gVisor implements a userspace kernel but does NOT enforce agent-specific privilege separation (all agent code, tool calls, and external network calls run in the same security domain). Model Armor for prompt injection defense. No ring separation.

**Gap:** A prompt-injected agent can: read session state (should be ring-1 read-only), execute any declared tool (should be ring-2, gated), make any network call the SA allows (should be ring-3, declared endpoints only). No syscall-level boundary between these three trust domains.

**Recommendation:** Implement 3-ring privilege model using gVisor seccomp profiles (Cloud Run already uses gVisor — this is policy configuration, not net-new infra):
- **Ring-1 (Agent Reasoning):** read-only access to session state + input context; no syscalls to tool layer
- **Ring-2 (Tool Execution):** controlled by Tool Access Manifest declared at agent registration; undeclared tool → syscall blocked → `SecurityAlert` emitted
- **Ring-3 (External Network):** egress only to endpoints declared in agent manifest; undeclared host → TCP blocked at gVisor layer
- Each ring boundary emits a structured log event to Cloud Logging: `{ring_crossing, agent_id, tool/endpoint, allowed: bool}`
- Violations terminate session + emit `agent.security.ring_violation` metric to Cloud Monitoring

**Impact:** Closes the ring-0 gap documented in today's paper. Makes GEAP the only agent platform with provable OS-style privilege separation. Required for FedRAMP High / HIPAA agent workloads. Differentiates from LangGraph, OpenAI SDK, CrewAI (none have ring-based models). gVisor is already in Cloud Run — this is ~2 sprint engineering effort, not infrastructure work.

**Priority:** HIGH — active exploitation in the wild; ring-0 gap documented in peer-reviewed literature today; gVisor substrate already exists

## Message 3 of 5

💡 **Agent Action Provenance Ledger API — Cryptographic Per-Action Audit Trail**

**Evidence:** [arXiv:2605.17169](https://arxiv.org/abs/2605.17169) (Hu et al., today) — "Every agent action must carry cryptographic provenance: model ID, tool version, instruction source, session ID, timestamp, hash of input context. Without this, agent-involved incidents cannot be audited or liability attributed at scale." Cloud Audit Logs capture API calls but NOT the causal chain of agent reasoning → instruction → tool invocation. No competitor has shipped this yet.

**GCP Today:** Cloud Audit Logs capture API calls (method, resource, identity, timestamp). Agent Runtime observability shows token/latency/error metrics. ZERO coverage of: which instruction triggered which tool call, whether that instruction was injected, what the agent's reasoning state was, whether the action was authorized by the original user task or a downstream tool output.

**Gap:** When an enterprise agent incident occurs (wrong action taken, data leaked, extortion attack like May 27), GCP customers cannot reconstruct the causal chain. They have "agent X called API Y at time Z" but not "because injected instruction W from tool output V triggered it." This blocks regulated industry adoption at the liability layer.

**Recommendation:** New resource type: `AgentProvenanceRecord` (auto-emitted per agent action):
```json
{
  "record_id": "apr_...",
  "session_id": "agt_sess_...",
  "action_id": "action_042",
  "parent_action_id": "action_039",
  "timestamp": "2026-05-28T20:31:00Z",
  "model_id": "gemini-3-5-flash@2026-05-01",
  "tool_name": "web_search",
  "tool_version": "v2.3.1",
  "instruction_source": "user | memory | tool_output | injection_flagged",
  "context_hash": "sha256:abc123...",
  "response_hash": "sha256:def456...",
  "ring_level": 2
}
```
Store in Cloud Spanner (immutable append-only, per-session partition). Expose via:
- `gcloud agents provenance list --session SESSION_ID` (causal chain reconstruction)
- Cloud Logging export (SIEM integration)
- Optional: sign records with Cloud HSM key (tamper-evident for FedRAMP/SOC2)
- GEAP console: "Agent Incident Timeline" view for selected session

**Impact:** First-mover in agent-action auditability — no AWS/Azure equivalent announced. Prerequisite for agent liability attribution in legal/financial/healthcare verticals. Direct enterprise procurement requirement (FINRA rule 17a-4 equivalent for AI agents is coming). Cloud Spanner infra already exists — this is schema + API surface work.

**Priority:** HIGH — first-mover opportunity; regulated vertical adoption gated on this; peer-reviewed requirement published today

## Message 4 of 5

💡 **ADK Coordination Layer Primitives — Policy-First Multi-Agent Orchestration**

**Evidence:** [arXiv:2605.03310](https://arxiv.org/abs/2605.03310) (Hu et al., today) — "Multi-agent production failure rates are 41-87%. Root cause: coordination failures, not model failures." Four required coordination primitives documented: routing (which agent handles which task class), delegation (when/how to hand off with budget), aggregation (how to combine parallel results), termination (provable exit condition). ADK has `ParallelAgent`, `SequentialAgent`, `LoopAgent` but no coordination policy layer above them.

**GCP Today:** ADK v2 provides agent type primitives. `adk deploy` CLI. Agent Studio visual topology. No static topology validation, no policy constraints, no coordination contract enforcement. Developers implement all coordination logic ad-hoc → fragile → fail in production.

**Gap:** ADK gives you the agents but not the contract between them. Every team hand-rolls routing/aggregation/termination differently. The paper's 41-87% failure rate is the consequence. No competitor (LangGraph, OpenAI Agents SDK, CrewAI) has solved this — GCP can lead with a coordination-as-first-class-concept model.

**Recommendation:** Add `CoordinationPolicy` resource to ADK v2.x:

```python
from google.cloud.adk import CoordinationPolicy, RoutingPolicy, AggregationStrategy

policy = CoordinationPolicy(
    routing=RoutingPolicy.rules([
        ("task_type == 'research'", researcher_agent),
        ("task_type == 'code'", coder_agent),
    ]),
    delegation_budget=DelegationBudget(max_hops=3, max_cost_usd=0.50),
    aggregation=AggregationStrategy.MAJORITY_VOTE,  # or WEIGHTED_MERGE | FIRST_WINS | CUSTOM
    termination=TerminationCondition.all_of([
        TerminationCondition.explicit_done_signal(),  # non-LLM-gated exit required
        TerminationCondition.max_iterations(20),
    ])
)
```

CLI validator: `adk topology-check --agent MyCompositeAgent` — enumerates topology, flags known failure modes:
- Relay chain > 3 hops → WARNING (relay drift risk)
- ParallelAgent with overlapping output keys → ERROR (write-set collision)  
- LoopAgent with only LLM-gated termination → ERROR (unprovable exit)

Visualize coordination graph with failure-mode annotations in Agent Studio.

**Impact:** Reduces the 41-87% coordination failure rate with research-backed primitives. Positions ADK as "the framework that validates your multi-agent topology before production." Differentiates from every competitor framework. Reduces customer support incidents from broken agent topologies. Paper-to-product turnaround signals GCP platform team reads and applies current research.

**Priority:** HIGH — 41-87% failure rate is a retention/satisfaction crisis; ADK v2.x patch scope (not a rewrite)

💡 **GEAP Agent-Native Pricing (AgentComputeUnit) — Workload-Aligned Billing**

**Evidence:** [arXiv:2605.26297](https://arxiv.org/abs/2605.26297) (Yuan et al., today) — "First empirical characterization of agentic workloads: 3-8x higher memory pressure, bursty compute, longer tail latency vs. batch inference." Current GEAP/Gemini API pricing is per-token, tuned for batch. Agents are a different workload class. AWS AgentCore charges per-microVM-second (compute-accurate for bursty workloads). Azure Foundry charges per-session. GCP charges per-token regardless of agent topology complexity, session duration, or memory burst.

**GCP Today:** Per-token pricing across Gemini API + GEAP. Agent Engine sessions billed by token consumption. No session duration factor. No memory burst factor. No tool invocation cost. Result: a 7-day long-running agent and a single-turn agent pay the same per token, despite 10x different infrastructure cost profiles.

**Gap:** Per-token pricing is wrong for agents in two ways: (1) undercharges for memory-intensive long sessions (loss leader as agents scale) and (2) creates sticker shock for short tasks with many small tool calls. Both hurt enterprise adoption — finance teams can't model costs predictably with per-token billing for autonomous agents.

**Recommendation:** Introduce `AgentComputeUnit (ACU)` as GEAP Agent Engine pricing unit:
- `ACU = base_tokens × topology_multiplier × session_duration_factor × memory_burst_factor`
- Topology multipliers: SingleAgent=1.0x, ParallelAgent=1.2x, MultiHop=1.5x (reflects coordination overhead)
- Session duration factor: 0-1h=1.0x, 1-4h=0.9x, 4-24h=0.8x, 24h+=0.7x (rewards long-running efficiency)
- Memory burst factor: auto-detected from workload profiler; nominal=1.0x, 3x burst=1.2x, 8x burst=1.5x
- Pricing tiers: Free (1,000 ACU/month), Standard ($0.002/ACU), Enterprise (committed ACU pools + SLA)
- GEAP console: ACU cost estimator tool — paste agent config → get predicted ACU/task + monthly projection
- Billing export: ACU breakdown per agent identity in Cloud Billing

**Impact:** Accurate pricing = removes enterprise procurement hesitation (predictable cost modeling). Free ACU tier = better acquisition than per-token free tier (customers can run real agents, not just test prompts). Committed ACU pools = enterprise contract mechanism. Positions GCP as "the cloud that prices for how agents actually work." Research-backed pricing change — cite the workload paper directly in the pricing docs.

**Priority:** MEDIUM — revenue/pricing change requires GTM coordination; but the research case is airtight today; start with free tier ACU as a quick win
