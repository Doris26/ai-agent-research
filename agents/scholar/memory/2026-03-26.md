#### Q1: RL-Based Tool-Use Reliability (Hermes Agent / Atropos)

**Key Finding: DeepHermes ToolCalling Specialist — first published RL-vs-baseline tool-calling comparison**
- Model: DeepHermes 3 Llama-3.1 8B fine-tuned with Atropos RL
- Berkeley Function Calling Benchmark results:
  - **Parallel: 0.10 → 0.46 (4.6x improvement)**
  - **Simple: 0.21 → 0.52 (2.5x improvement)**
- This is the first published evidence of RL dramatically outperforming base models on structured tool-calling during reasoning
- Atropos itself is purely engineering (no arXiv paper) — it's an async environment microservice framework for distributed LLM RL rollouts. Blog post: "Atropos: An Async First Environment Rollout Controller" (Mahan, Jin, Teknium et al., 2025)

**Related Academic Papers:**
- **Agent-R1** (2511.14460) — End-to-end RL framework for LLM agents extending MDP formulation. Modular, validated on Multihop QA.
- **Memory-R1** (2508.19828) — RL for memory management in agents. Complements Atropos tool-calling RL.
- **Towards Verifiably Safe Tool Use** (2601.08012) — Formal verification for safe tool use (complements RL reliability with guarantees).
- **AgentRM** (already in ledger) — Process-level reward models for step-wise agent guidance.

**ADK Implication:** The 4.6x improvement on parallel tool-calling is massive. If RL-tuned models dramatically outperform prompt-tuned on tool reliability, GCP needs a fine-tuning story. Current ADK relies on prompt engineering for tool-calling — Atropos-style RL fine-tuning could be a competitive moat.

**Is Atropos documented academically?** No — purely engineering-side with GitHub docs + blog post. No peer-reviewed paper. This is a gap: the most impactful RL-for-tool-calling work is undocumented in academic literature.

#### Q2: Hierarchical SuperAgent Architectures (DeerFlow 2.0)

**Key Papers Found:**
- **HALO** (2505.13516) — 3-tier hierarchy: planning → role-design → inference agents. MCTS for action space search. **14.4% avg improvement over flat baselines, up to 19.6% on MATH.** Closest academic equivalent to DeerFlow's architecture.
- **Deep Agent** (2502.07056) — Recursive planner-executor on Hierarchical Task DAG (HTDAG). Multi-layer decomposition for extended autonomous operations. Most direct match to DeerFlow's hours-long task pattern.
- **Plan-and-Act** (2503.09572) — Explicit plan-then-execute pipeline for long-horizon tasks. Addresses reactive prompting limitations.
- **MiRA** (2603.19685, already in ledger) — Milestone-based rewards for long-horizon web agents.

**Long-Horizon Duration Data:** AI task duration doubling every 7 months. Currently ~2-hour autonomous operations. Performance degradation observed after ~35 minutes of equivalent human time. DeerFlow 2.0's "hours-long" claim is at the frontier of what's achievable.

#### Q3: Agent Portability Specs (GitAgent / "Docker for Agents")

**Key Papers Found:**
- **Open Agent Spec** (2510.04173) — **First formal agent portability standard.** Declarative language, 4 runtimes (LangGraph, CrewAI, AutoGen, WayFlow), 3 benchmarks. PyAgentSpec SDK. Define-once, run-anywhere for agents.
- **Docker cagent** — Docker's own open-source agent framework. Vendor-agnostic. Portability + execution speed.
- **GitAgent** — Framework-agnostic spec: agent as structured Git directory. Export to LangGraph/AutoGen/Claude Code. "Universal Format."
- **AgenticFormat Standard** — YAML/JSON declarative schema separating Cognitive Blueprint from Runtime Engine.

**Convergence Pattern:** Three independent efforts (Agent Spec academic, Docker cagent industry, GitAgent open-source) all converging on the same pattern: declarative agent definition decoupled from execution runtime. This is the "Dockerfile for agents" moment.

### Research Questions: Computer Use + Edge Agent Runtimes

#### Q1: Computer Use / Browser Automation Academic Foundation (Azure Foundry context)

**Desktop UI Control Agents — Key Architecture Papers:**

1. **UFO2: The Desktop AgentOS** (2504.14603, Microsoft Research, 20+ authors) — **THE reference paper for Azure Computer Use.** Multi-agent OS for Windows: HostAgent for task decomposition + specialized AppAgents per application. **Hybrid GUI+API execution** = Windows UI Automation (UIA) + vision-based parsing combined. PiP virtual desktop for safety isolation. Speculative multi-action planning. This IS the academic foundation for Azure's managed Computer Use service.

2. **GUI-R1** (2504.10458) — RL framework enhancing VLM GUI capabilities across ALL platforms (Windows/Linux/macOS/Android/Web). Unified action space modeling. Cross-platform agent architecture.

3. **ScreenAgent** (2402.07945) — Earlier vision-language computer control agent (IJCAI-24). Established screenshot-to-action paradigm.

4. **UI-TARS-2** (2509.02544) — Multi-turn RL for GUI agents. Reproducible data pipeline spanning mainstream websites + desktop apps.

**Key Architectural Difference: Desktop vs Web agents:**
- Web agents: access DOM/HTML → structured data → action. Vision optional.
- Desktop agents: NO structured data for arbitrary apps → must use vision + OS accessibility APIs. UFO2's hybrid UIA+vision is the state of the art.
- Desktop is strictly harder: larger action space, no DOM, heterogeneous app UIs.

**Benchmarks:**
- **OSWorld** (2404.07972) — 369 real tasks across Ubuntu/Windows/macOS. Human: 72.36%, **SOTA now 69.9%** (approaching human). The canonical benchmark.
- **WindowsAgentArena** (2409.08264, Microsoft) — 150+ Windows tasks. Parallelizable in Azure (20 min full eval). **Microsoft built their own benchmark for this managed service.**

**Playwright vs Vision-Model Control:**
- Playwright/scripted: <2 seconds per task, deterministic, breaks on layout changes. $0 inference cost.
- Vision model (BrowserUse): 15-30 seconds, resilient to UI changes, $0.01-0.05+ per task.
- **No rigorous academic benchmark directly comparing them.** Industry comparisons only (Stagehand, BrowserUse). This is a gap.

**Safety/Isolation Patterns for Enterprise:**
- **Fault-Tolerant Sandboxing** (2512.12806) — Transactional filesystem: 100% interception, 100% rollback, 14.5% overhead. Best published enterprise sandbox pattern.
- **UFO2 PiP isolation** — Virtual desktop isolation for concurrent user+agent operations.
- **Building Browser Agents** survey (2511.19477) — Covers sandbox, credential handling, enterprise patterns.
- **2025 AI Agent Index** (2602.17753) — Only 9/30 deployed agents have sandboxing. Huge deployment gap.

#### Q2: Edge/Microcontroller Agent Runtimes (NullClaw/KrillClaw context)

**This is a REAL research area, not niche. Multiple paper clusters:**

**Edge Agent Architecture (top papers):**

1. **Edge General Intelligence with Agentic AI** (2508.18725) — Maps agentic AI to 6G/IoT edge. Perception-reasoning-action loops under resource constraints. Compact compression + energy-aware computing. THE survey paper for edge agents.

2. **Collaborative Edge SLMs + Cloud LLMs** (2507.16731) — Survey of edge-cloud collaboration: task assignment (what runs where), task division (splitting computation), speculative decoding, modular routing at task AND token level. **Core architecture for NullClaw-style edge inference + cloud planning.**

3. **Splitwise** (2512.23310) — Lyapunov DRL for fine-grained LLM partitioning across edge/cloud. Decomposes into attention heads + FFN sub-blocks. Jointly optimizes latency + energy + accuracy.

**Energy/Latency Tradeoffs:**

4. **Sustainable LLM Inference for Edge AI** (2504.03360) — 28 quantized LLMs on Raspberry Pi 4. **Quantization: 69% latency reduction, 79% energy reduction vs FP16.** Diminishing returns at lower precision.

5. **Camel** (2508.09173) — First latency-vs-energy tradeoff model for LLM batching on constrained devices. MAB optimization. 12.4-29.9% energy-delay-product reduction.

6. **LLM Inference at the Edge** (2603.23640, March 2026) — Mobile/NPU/GPU trade-offs under sustained load. Most recent paper on always-on personal agents.

**TinyML / Sub-MB Connection:**

7. **TinyLLM** (2412.15304) — 30M-124M parameter GPT-2 models at edge. Framework for edge training + deployment.

8. **Skilled AI Agents for Embedded/IoT** (2603.19583) — LLM agents for hardware-in-the-loop. 378 experiments across 3 platforms. Key finding: code compiles ≠ works on device. Human-expert skills essential.

**Key Insight for GCP:** The edge-cloud split architecture is well-documented academically. The pattern is:
- Edge: SLM (sub-3B) handles perception + simple actions + latency-critical responses
- Cloud: LLM handles planning + complex reasoning + memory management
- Communication: task assignment + speculative decoding + modular routing

NullClaw (678KB) and KrillClaw (49KB) are at the extreme end — sub-MB. Academic literature covers down to Raspberry Pi 4 (30M-124M params). The $3 microcontroller tier is genuinely novel and has no direct academic coverage.
